スタートアップ

RHACS Operator と考えて ACS をインストールし、その後、ワークショップにサンプルを提供する小さなデモをインストールします。

RHACS オペレーターのインストール

  1. Operator Hub から Advanced Cluster Security Operator を検索します。

  2. Installボタンをクリックして、選択したオペレーターをインストールします。

  3. インストール パラメータの確認

    • rhacs-3.70 チャネル

    • rhacs-operator namespace

    • Automatic アップデート

  4. 完了するまで待ちます。インストールには数分かかります。

  5. View Operator ボタンをクリックして、準備完了のオペレーターにアクセスしView Operatorます。

RHACS Central Cluster のインストール

このセクションでは、Central コンポーネントをラボ クラスタにデプロイします。Central は、次の 2 つの主要なデプロイメントで構成されています。

  • The central serviceは、API とコンソールを公開し、セキュリティで保護されたクラスター上のセンサーと通信します。

  • The scanner serviceは、デプロイされたPodsのイメージをスキャンする役割を持ちます。

OpenShift クラスターにログインし、新しい stackrox namespace を作成します (次のように Web コンソールまたは cli を使用します)。ここにコンポーネントをインストールします。

oc new-project stackrox

次に、ACS Central で管理者ユーザーが使用するパスワードを使用してSecretを作成します。このSecretは、後で ACS Central を作成するときに指定します。
これは必須の作業ではありませんが、このワークショップの後のラボのために、acs-passwordをSecretにして管理者パスワードとしてstackroxを設定する必要があります。

Secret を指定せずに Central を作成すると、管理者ユーザーのパスワードとしてランダムな文字列が設定されます。 
oc create secret generic acs-password --from-literal=password=stackrox -n stackrox

次に、Central を作成しましょう。以下は、中央のカスタム リソースの例です。これは、Web コンソールまたは CLI の 2 つの方法で行うことができます。

apiVersion: platform.stackrox.io/v1alpha1
kind: Central
metadata:
  name: stackrox-central-services
  namespace: stackrox
spec:
  central:
    exposure:
      loadBalancer:
        enabled: false
        port: 443
      nodePort:
        enabled: false
      route:
        enabled: true
    adminPasswordSecret:
      name: acs-password
    persistence:
      persistentVolumeClaim:
        claimName: stackrox-db
  egress:
    connectivityPolicy: Online
  scanner:
    analyzer:
      scaling:
        autoScaling: Enabled
        maxReplicas: 5
        minReplicas: 2
        replicas: 3
    scannerComponent: Enabled

Web コンソールで

  1. Operator準備完了画面から View Operator をクリックするか、Operator → インストール済の Operator ページに移動します。次に、Provided APIセクションで、Central APIのCentralの作成を選択します。

  2. YAMLビュー ラジオボタンをチェックし、下に表示されているように Central のカスタムリソースの例 を貼り付けて Create ボタンをクリックします。

  3. Topology メニューの Developer パースペクティブに切り替えることで、進捗状況を確認することができます。

Developer パースペクティブを使用して同じことを行うこともできます。

  1. 選択した stackrox プロジェクト内で +追加 → YAMLのインポートを選択

  2. YAML コンテンツを貼り付けて、 Create ボタンをクリックします。

OCクライアントで

  1. Create the このリポジトリで提供されるテンプレート ファイルを使用して、 central カスタムリソースを作成します。

    oc apply -f stackrox-central-services.yaml -n stackrox

  2. watch オプションを使用してインストールを監視します。

    oc get pods -n stackrox -w

RHACS ログイン

  1. インストールが完了したら、次のようにコマンド ラインから、または Topology Web コンソールのビューを使用して、生成されたルートのホスト名を抽出します。

    oc get routes/central -n stackrox -o jsonpath='{.spec.host}'

  2. Login to https://<route_hostname>; にログインします。ユーザー名 admin とパスワード stackrox を使用します。

RHACS セキュア クラスタ構成

クラスターを ACS にインポートするには、Sensor、Collector、Admission Controller の TLS シークレットを含むクラスター初期化バンドルを生成する必要があります。

RHACS ポータルを使用した init バンドルの生成

  1. Platform Configuration セクションの Integration サブセクションにアクセスして、クラスター初期化バンドルを生成します。

  2. 一意のクラスタ名（ここではdemo-cluster）を持つバンドルを生成します。

  3. クラスター初期化バンドル シークレットをダウンロードします。

  4. ターゲット セキュア クラスタにクラスタ初期化バンドル シークレットを適用します

    oc apply -f ~/Downloads/demo-cluster-cluster-init-secrets.yaml -n stackrox

    +Add → Import YAML ショートカットを使用して、yaml ファイルを開き、コンテンツをコピーして Web コンソールに貼り付けることができます。すべて cluster-admin のオブジェクトに namespace キーが設定されているため、Web コンソールで選択したプロジェクト (すべてのプロジェクトなど) に関係なく、それぞれのnamespaceで作成されたすべてのオブジェクトを取得します。

セキュア クラスタ サービスのインストール

このワークショップでは、CentralおよびSecured Clusterと同じクラスターを使用します。リアルタイムのシナリオでは、さまざまなSecured Clusterが多数存在します。SecuredCluster カスタムリソース を管理するために、すべてのSecured Clusterに ACS Operator をインストールしてください。

The SecuredCluster カスタムリソースは非常に単純です。 次の例では demo-cluster ターゲットの構成を示しています。 collector 収集方法が EBPF に設定されている構成に注意してください。別の収集方法として KernelModule があります。TolerateTaintsにより、Collectorデーモンセットは、ODFノードのような特殊なテイントを持つノードにもデプロイされます。

apiVersion: platform.stackrox.io/v1alpha1
kind: SecuredCluster
metadata:
  name: stackrox-secured-cluster-services
  namespace: stackrox
spec:
  admissionControl:
    listenOnCreates: true
    listenOnEvents: true
    listenOnUpdates: true
  clusterName: demo-cluster
  perNode:
    collector:
      collection: EBPF
      imageFlavor: Regular
    taintToleration: TolerateTaints
詳細については、SecuredCluster オペレーターのドキュメント の設定を確認してください。

  1. リポジトリで提供されている例を使用して (およびオプションでカスタマイズして)、Secured Cluster カスタム リソースを作成します。

    oc apply -f stackrox-secured-cluster-services.yaml -n stackrox

    または、Web コンソールを使用して、ACS Operator ビューで次のようにします。

    1. 提供された API セクションで、SecuredClusrerの作成 を選択します。

    2. そして、yamlコンテンツをコピーして貼り付けます

    または、前のセクションのように、+Add → Import YAML パスを使用します。

  2. ウォッチオプションを使用してインストールを監視します（または、前述のようにDeveloper視点でウェブコンソールのTopologyビューを使用します）。

    oc get pods -n stackrox -w

  3. インストールの最後に、Central コンソールに移動し、Secured Clusterが正しく接続されていることを確認します。

RHACS でのデモの展開 - 必須

使用したインストール オプションに関係なく、ACS デモをクラスタに展開する必要があります。

  1. デモを含むレポをダウンロードします。

    sudo dnf install -y ansible
ansible-galaxy collection install kubernetes.core
sudo pip3 install kubernetes openshift jmespath
cd
git clone https://github.com/rh-mobb/rhacs-demo
cd rhacs-demo

  2. ansible デモをクラスターに適用します。

    Playbook を実行する前に、OpenShift クラスターにログインする必要があります。 
    ansible-playbook rhacs-demo.yaml -e stackrox_central_admin_password=stackrox

  3. Playbook の実行後 (すべてが正常に機能した場合)、出力は次のようになります。

    TASK [ocp4_deploy_acs_demo_apps : Lock baselines] ****************************************************************************************
ok: [localhost] => (item=visa-processor)
ok: [localhost] => (item=backend-atlas)
ok: [localhost] => (item=asset-cache)

PLAY RECAP *******************************************************************************************************************************
localhost                  : ok=35   changed=3    unreachable=0    failed=0    skipped=0    rescued=0    ignored=0

アプリのデプロイ - オプション

残りのセクション（デモのデプロイに基づくもの）に従う予定がなく、遊びで独自の発見をしたい場合は、以下の手順に従ってください。

  1. 重大な脆弱性があるいくつかのアプリケーションを開始します。

    oc new-project test
    oc run shell --labels=app=shellshock,team=test-team \
  --image=vulnerables/cve-2014-6271 -n test
    oc run samba --labels=app=rce \
  --image=vulnerables/cve-2017-7494 -n test

  2. RHACS ポータルに移動して、違反を表示します。
